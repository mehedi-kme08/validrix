## What is Validrix?

Validrix is a pytest plugin framework that brings AI into every stage of your test lifecycle. It solves the three biggest pain points in test automation:

| Pain | How Validrix fixes it |
|------|----------------------|
| **Writing tests is slow** | Describe a feature in English — Validrix generates the full pytest file |
| **Flaky tests erode CI trust** | Automatically detects intermittent failures by running each test N times |
| **Locator maintenance is endless** | Self-healing Playwright selectors recover automatically when the UI changes |
| **Failures are hard to diagnose** | AI reads all failures together and explains root causes in plain English |
| **Testing a website from scratch** | Give it a URL — it crawls, generates tests, runs them, and reports results |

### 1. AI Test Generator

**What it does:** Converts a plain-English description of a feature or page into a complete, runnable `pytest` file.

**When to use it:** Starting a new feature, onboarding to an unfamiliar codebase, or eliminating boilerplate test writing.

#### Via CLI

```bash
# Generate and print (dry run)
validrix generate "Login page with email, password, and Remember Me checkbox" --dry-run

# Generate and save to a file
validrix generate "User registration with name, email, and password validation" \
  --output tests/test_registration.py

# Use OpenAI instead of Anthropic
validrix generate "Checkout flow with Stripe payment" \
  --provider openai \
  --output tests/test_checkout.py

# Add extra context to guide generation
validrix generate "Product search page" \
  --context "Use pytest.mark.parametrize for search terms. The search bar has id='q'." \
  --output tests/test_search.py
```

#### Via Python API

```python
from validrix.plugins.ai_generator import AITestGenerator
from pathlib import Path

gen = AITestGenerator()

result = gen.generate(
    description="Login page with email, password, and Remember Me checkbox",
    output_path=Path("tests/test_login.py"),
    extra_context="Use Playwright page fixture. Base URL is http://localhost:3000",
)

print(result.code)          # the generated Python source
print(result.saved_path)    # Path("tests/test_login.py")
print(result.token_usage)   # {"input_tokens": 412, "output_tokens": 890}
```

#### Example output

```python
# AUTO-GENERATED by Validrix AI Generator
# Feature: Login page with email, password, and Remember Me checkbox

import pytest
from playwright.sync_api import Page, expect

class TestLoginPage:

    @pytest.mark.smoke
    def test_valid_login_redirects_to_dashboard(self, page: Page) -> None:
        """Valid credentials should redirect to the dashboard."""
        page.goto("/login")
        page.locator("#email").fill("user@example.com")
        page.locator("#password").fill("SecurePass123!")
        page.locator("button[type='submit']").click()
        expect(page).to_have_url("/dashboard")

    @pytest.mark.parametrize("email,password,expected_error", [
        ("", "SecurePass123!", "Email is required"),
        ("not-an-email", "pass", "Invalid email format"),
        ("user@example.com", "", "Password is required"),
    ])
    @pytest.mark.regression
    def test_invalid_inputs_show_errors(
        self, page: Page, email: str, password: str, expected_error: str
    ) -> None:
        """Invalid inputs should surface a specific validation error."""
        page.goto("/login")
        page.locator("#email").fill(email)
        page.locator("#password").fill(password)
        page.locator("button[type='submit']").click()
        expect(page.locator(".error-message")).to_contain_text(expected_error)

    def test_remember_me_checkbox_is_visible(self, page: Page) -> None:
        """Remember Me checkbox must be present and interactive."""
        page.goto("/login")
        checkbox = page.locator("input[type='checkbox'][name='remember']")
        expect(checkbox).to_be_visible()
        checkbox.check()
        expect(checkbox).to_be_checked()
```

### 2. Self-Healing Locators

**What it does:** Wraps Playwright's `Page` object so that when a CSS selector fails, Validrix automatically tries a chain of fallback strategies before the test fails. Every healing event is logged to `healing_history.json`.

**When to use it:** UI tests that break every sprint when designers rename elements, change IDs, or restructure the DOM.

#### Setup — use the `healing_page` fixture

No code changes needed. Install Validrix and the fixture is auto-registered. Just use `healing_page` instead of `page`:

```python
# conftest.py — nothing needed, fixture is auto-provided by Validrix

# test_checkout.py
from playwright.sync_api import Page

def test_complete_checkout(healing_page) -> None:
    """healing_page is a drop-in replacement for page with auto-healing."""
    healing_page.goto("http://localhost:3000/checkout")

    # If "#submit-btn" is renamed to "#place-order" in the next sprint,
    # Validrix heals it automatically using aria-label or text content.
    healing_page.locator("#submit-btn").click()

    assert healing_page.url.endswith("/confirmation")
```

#### Fallback chain (configurable)

```
Original selector fails
        │
        ▼
1. aria-label  →  button[aria-label="Submit Order"]
        │ fails
        ▼
2. text        →  text="Submit Order"
        │ fails
        ▼
3. nearby      →  ancestor/sibling search around original position
        │ fails
        ▼
4. css rebuild →  broad selector rebuild from tag + attributes
```

#### Reading the healing history

```python
import json
from pathlib import Path

history = json.loads(Path("validrix_reports/healing_history.json").read_text())

print(f"Total healed: {history['total_healed']}")
for event in history["events"]:
    print(f"{event['original_selector']} → {event['healed_selector']} via {event['successful_strategy']}")
```

#### Configure fallback order

```yaml
# validrix.yml
healing:
  enabled: true
  history_path: validrix_reports/healing_history.json
  fallback_order:
    - aria-label   # most stable (accessibility attribute)
    - text         # visible text — survives renames
    - nearby       # structural fallback
    - css          # last resort
```

### 3. Flaky Test Detector

**What it does:** Runs each test N times in a single session and computes a pass rate and flakiness score. Generates both a JSON report and a self-contained HTML dashboard.

**When to use it:** Before merging a branch to catch tests that pass 70% of the time, or as a nightly job to audit suite reliability.

#### Run from CLI

```bash
# Run all tests 5 times each
pytest --detect-flaky --flaky-runs=5

# Target a specific marker
pytest --detect-flaky --flaky-runs=3 -m smoke

# Override threshold (default 0.5 = tests below 50% pass rate are FLAKY)
pytest --detect-flaky --flaky-runs=5 --flaky-threshold=0.8
```

#### Terminal output

```
FLAKY   tests/e2e/test_payment.py::test_stripe_webhook   (60% pass  score: 0.80)
FLAKY   tests/e2e/test_search.py::test_autocomplete      (80% pass  score: 0.48)
STABLE  tests/unit/test_auth.py::test_token_expiry       (100% pass)
STABLE  tests/unit/test_user.py::test_create_user        (100% pass)
FAILING tests/e2e/test_cart.py::test_add_item            (0% pass)

Summary: 2 flaky / 1 failing / 2 stable  (5 runs each)
Reports → validrix_reports/flaky_report.json
         validrix_reports/flaky_report.html
```

#### Reading results in Python

```python
import json
from pathlib import Path

report = json.loads(Path("validrix_reports/flaky_report.json").read_text())

print(f"Flaky tests: {report['total_flaky']}")
print(f"Failing tests: {report['total_failing']}")

for result in report["results"]:
    if result["label"] == "FLAKY":
        print(f"{result['test_id']}: {result['pass_rate']:.0%} pass rate")
```

#### Flakiness score explained

The flakiness score peaks at 1.0 when a test passes exactly 50% of the time — maximally unreliable. It is 0.0 when a test always passes or always fails (deterministic in either direction).

```
score = 1 - |2 × pass_rate - 1|

Examples:
  pass_rate = 1.0  →  score = 0.0  (STABLE)
  pass_rate = 0.8  →  score = 0.4  (mild flakiness)
  pass_rate = 0.5  →  score = 1.0  (maximally flaky)
  pass_rate = 0.0  →  score = 0.0  (FAILING — deterministic)
```

### 4. AI Failure Summariser

**What it does:** After a test session ends, collects all failures and sends them in a single batch to Claude. Returns a Markdown root-cause analysis with severity ratings and suggested fixes. Works as a post-session pytest hook — zero extra commands needed.

**When to use it:** After a CI run fails, to instantly understand what broke and why without digging through 40 raw tracebacks.

#### Automatic — runs at end of every session

As long as your API key is set, the reporter activates automatically after any failing `pytest` run:

```bash
pytest tests/

# → After the run, Validrix writes:
#   validrix_reports/report.md
#   validrix_reports/report.json
```

#### Generate a report manually from a previous run

```bash
# Reads the failure checkpoint saved during the last session
validrix report

# Save to a custom directory
validrix report --output-dir ci_artifacts/
```

#### Python API

```python
from validrix.plugins.ai_reporter import AIReporterPlugin

reporter = AIReporterPlugin()
report_path = reporter.generate_from_checkpoint()
print(f"Report saved to: {report_path}")
```

#### Example report output

```markdown

All failures share the same `401 Unauthorized` response from `/api/v2/users`.
```

### 5. BaseTest + RetryManager

**What it does:** `BaseTest` is an optional abstract base class that gives your test classes typed config access, a pre-configured retry manager, soft assertions (collect multiple failures without stopping), and polling assertions.

**When to use it:** Teams that want a shared base class with consistent setup/teardown rather than composing individual fixtures.

#### Extending BaseTest

```python
from validrix.core.base_test import BaseTest
import pytest

class TestUserProfile(BaseTest):

    def setup_method(self, method):
        super().setup_method(method)
        # self.config, self.retry, self.log all available here
        self.base_url = self.config.env.base_url

    def test_profile_page_loads(self) -> None:
        """User profile page should return 200 within 3 retries."""
        import httpx

        # Retry up to config.retry.max_attempts times with exponential backoff
        response = self.retry.execute(
            lambda: httpx.get(f"{self.base_url}/api/profile", timeout=5)
        )
        assert response.status_code == 200

    def test_profile_fields_are_complete(self) -> None:
        """Profile must have name, email, and avatar — collect all failures."""
        import httpx
        data = httpx.get(f"{self.base_url}/api/profile").json()

        # soft_assert collects failures without stopping the test
        self.soft_assert("name" in data, "Profile missing 'name' field")
        self.soft_assert("email" in data, "Profile missing 'email' field")
        self.soft_assert("avatar_url" in data, "Profile missing 'avatar_url' field")

        # Raises if any soft assertion failed, showing ALL failures at once
        self.assert_soft_failures()

    def test_search_index_eventually_updates(self) -> None:
        """After a write, the search index should reflect it within 10 seconds."""
        import httpx

        # Polls condition every 0.5s until True or 10s timeout
        self.assert_eventually(
            condition=lambda: "new-user" in httpx.get(f"{self.base_url}/api/search").text,
            timeout_seconds=10,
            message="Search index did not update within 10 seconds",
        )
```

#### Using RetryManager standalone

```python
from validrix.core.retry_manager import RetryManager, RetryConfig, retry
import httpx

# Option 1: decorator
@retry(max_attempts=5, delay_seconds=0.5, jitter=True)
def fetch_order_status(order_id: str) -> dict:
    return httpx.get(f"/api/orders/{order_id}").json()

# Option 2: RetryManager instance
policy = RetryConfig(max_attempts=3, delay_seconds=1.0, backoff_multiplier=2.0)
manager = RetryManager(policy)
result = manager.execute(lambda: httpx.get("/api/health").json())

# Option 3: custom config per call
@retry(config=RetryConfig(max_attempts=10, delay_seconds=0.2))
def poll_background_job(job_id: str) -> str:
    status = httpx.get(f"/api/jobs/{job_id}").json()["status"]
    if status == "pending":
        raise RuntimeError("Not done yet")
    return status
```

## 🌐 Web Agent

**What it does:** Give Validrix a live website URL and a plain-English description of what to test. It crawls the page with Playwright, sends the structure to Claude to generate a pytest test suite, executes those tests in headless Chromium, and produces an AI-written HTML report — fully automated, zero manual test writing.

**When to use it:** Auditing a third-party site, smoke-testing a deployment, onboarding to an unfamiliar frontend, or demonstrating test automation in a demo.

### How it works

```
URL + Prompt
     │
     ▼
┌──────────────┐   ┌───────────────────┐   ┌────────────────┐   ┌──────────────┐
│  WebCrawler  │──▶│  WebTestGenerator │──▶│  TestExecutor  │──▶│  WebReporter │
│ (Playwright) │   │   (Claude API)    │   │    (pytest)    │   │ (AI + HTML)  │
└──────────────┘   └───────────────────┘   └────────────────┘   └──────────────┘
  Extracts title,    Turns crawl data         Runs tests in         AI summary +
  headings, forms,   + your prompt into       headless Chromium,    dark-theme
  buttons, links     a full .py test file     captures screenshots  HTML report
```

### Option A — One-liner Python API

```python
from validrix.web_agent import run_pipeline
from pathlib import Path

result = run_pipeline(
    url="https://mhdhasan.netlify.app/",
    prompt=(
        "Verify the page title loads correctly. "
        "Check all navigation links are present and visible. "
        "Confirm the hero heading contains text. "
        "Test the contact section is reachable."
    ),
    report_dir=Path("validrix_reports"),
    max_tests=10,
    timeout_seconds=45,
    headless=True,
)

print(f"Health   : {result.health_label}")           # healthy | degraded | critical
print(f"Results  : {result.passed}/{result.total_tests} passed")
print(f"Report   : {result.report_path}")
print(f"\nAI Summary:\n{result.ai_summary}")
```

| Method | Endpoint | Description |
|--------|----------|-------------|

## ⚙ Configuration

All configuration lives in `validrix.yml`. Environment variables override YAML values using the prefix `VALIDRIX_` with `__` as the nested delimiter.

```yaml
# validrix.yml — complete reference

environment: dev              # which environment block is active

ai:
  provider: anthropic         # anthropic | openai
  model: claude-sonnet-4-20250514
  max_tokens: 4096
  temperature: 0.2            # lower = more deterministic output
  timeout_seconds: 60

retry:
  max_attempts: 3
  delay_seconds: 1.0
  backoff_multiplier: 2.0     # delay doubles each attempt
  jitter: true                # randomises delay to avoid thundering herd

flaky:
  enabled: true
  runs: 3                     # re-run count per test
  threshold: 0.5              # pass rate below this → labelled FLAKY
  report_path: validrix_reports/flaky_report.json

healing:
  enabled: true
  history_path: validrix_reports/healing_history.json
  fallback_order:
    - aria-label
    - text
    - nearby
    - css

environments:
  dev:
    base_url: http://localhost:3000
    headless: true
    timeout_ms: 30000
  staging:
    base_url: https://staging.example.com
    headless: true
    timeout_ms: 15000
  prod:
    base_url: https://example.com
    headless: true
    timeout_ms: 10000
```

**Environment variable overrides:**


## 🏗 Architecture

Validrix is organised into four independent layers. Each layer has a single responsibility and the lower layers have no knowledge of the layers above them.

```
┌─────────────────────────────────────────────────────┐
│  CLI  (validrix generate / run / report / scaffold) │
└────────────────────┬────────────────────────────────┘
                     │ calls
┌────────────────────▼────────────────────────────────┐
│  Plugins  (pytest hooks — auto-registered via       │
│           pyproject.toml entry points)              │
│                                                     │
│  ai_generator   flaky_detector   self_healing       │
│  ai_reporter    web_agent pipeline                  │
└────────────────────┬────────────────────────────────┘
                     │ depends on
┌────────────────────▼────────────────────────────────┐
│  Core  (primitives — no business logic)             │
│                                                     │
│  ConfigManager   BaseTest   RetryManager            │
└────────────────────┬────────────────────────────────┘
                     │ delegates to
┌────────────────────▼────────────────────────────────┐
│  Integrations  (infrastructure adapters)            │
│                                                     │
│  DockerRunner   FastAPI (web_agent/api)             │
└─────────────────────────────────────────────────────┘
```

| Layer | Files | Responsibility |
|-------|-------|----------------|
| **Core** | `core/config_manager.py` `core/base_test.py` `core/retry_manager.py` | Typed config, base classes, retry primitives |
| **Plugins** | `plugins/ai_generator.py` `plugins/flaky_detector.py` `plugins/self_healing.py` `plugins/ai_reporter.py` | pytest hook implementations — zero cross-plugin imports |
| **Web Agent** | `web_agent/` `api/` `ui/` | URL-to-test pipeline, FastAPI server, browser UI |
| **CLI** | `cli/framework_cli.py` | Thin developer UX shell over Core + Plugins |
| **Integrations** | `integrations/docker_runner.py` | Infrastructure adapters (Docker Compose) |

```bash
# 1. Fork and clone
git clone https://github.com/mehedi-kme08/validrix.git
cd validrix

# 2. Install dev dependencies
pip install -e ".[dev]"
playwright install chromium
pre-commit install

pytest tests/ --tb=short
```

**Code standards:**
- Type hints on all function signatures
- Docstrings on all public classes and methods

- New plugins must have zero import-time dependency on other plugins
